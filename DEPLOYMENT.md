# Deployment Guide — Campus Placement API + Campus Nexus App

Everything below the "You do" markers needs your own accounts; the rest is
already configured in this repo.

## 1. Database — Neon (free Postgres)

**You do:**
1. Create an account at https://neon.tech (free tier is enough).
2. Create a project (e.g. `campus-placement`), region closest to your users.
3. Copy the **connection string** (looks like
   `postgresql://user:pass@ep-xxx.region.aws.neon.tech/neondb?sslmode=require`).

Migrations run automatically during the Render build
(`npx prisma migrate deploy`), so no manual schema setup is needed.

To seed the production DB with students/coordinators, run locally with the
Neon URL:

```powershell
$env:DATABASE_URL = "<neon-connection-string>"
npm run db:seed
```

> Replace `prisma/seed.ts` contents with your college's real student list
> (or import a CSV) before seeding production.

## 2. Email — Brevo (free SMTP for OTP)

**You do:**
1. Create an account at https://www.brevo.com (free: 300 emails/day).
2. Settings → SMTP & API → note the SMTP server, port, login and key.
3. Verify a sender address (your college email works).

Alternative: Gmail app password (Google Account → Security → 2FA →
App passwords) with `SMTP_HOST=smtp.gmail.com` — capped ~500 mails/day.

## 3. Server — Render

**You do:**
1. Push this backend folder to a GitHub repo.
2. Create an account at https://render.com → New → **Blueprint** →
   point it at the repo. Render reads `render.yaml` automatically.
3. In the service's Environment tab fill the `sync: false` variables:
   - `DATABASE_URL` — the Neon connection string
   - `SMTP_HOST` / `SMTP_USER` / `SMTP_PASS` / `SMTP_FROM` — from Brevo
   - `CORS_ORIGIN` — only needed for browser clients; the Flutter app
     (no Origin header) is always allowed
4. Deploy. First build takes a few minutes.

**Smoke test:**
```powershell
Invoke-RestMethod https://<your-service>.onrender.com/api/health
# then request a real OTP and check the email arrives:
Invoke-RestMethod -Uri "https://<your-service>.onrender.com/api/auth/student/request-otp" `
  -Method Post -ContentType "application/json" -Body '{"email":"<seeded-student-email>"}'
```

> Free tier note: the service sleeps after 15 min idle (first request then
> takes ~50s) and cron deadline-reminders won't fire while asleep. The $7
> "starter" plan keeps it always-on — worth it for a real rollout.

## 4. Flutter release build

Once the backend URL is live:

```powershell
cd "..\ascent_campus"
flutter build apk --release --dart-define=API_BASE_URL=https://<your-service>.onrender.com/api
```

Output: `build\app\outputs\flutter-apk\app-release.apk` — signed with the
release keystore in `android/` (see `android/key.properties`; keep the
keystore and passwords backed up somewhere safe — losing them means you
can never update the app under the same signature).

Distribute the APK directly, via Firebase App Distribution, or publish to
Play Store ($25 one-time developer fee + review).

## Production checklist

- [ ] Neon DB created, `DATABASE_URL` set on Render
- [ ] Migrations applied (automatic in build) and real students seeded
- [ ] Brevo/Gmail SMTP vars set — OTP email actually received
- [ ] `JWT_SECRET` generated by Render (never reuse the dev one)
- [ ] Coordinator account created with a strong password (not admin123)
- [ ] `/api/health` returns ok from the public URL
- [ ] APK built with `--dart-define=API_BASE_URL=https://…` and login
      works on a real phone
